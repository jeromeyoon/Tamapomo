# MVP0.5 비판적 검토 대화

## 범위

MVP0.5는 MVP0의 가장 큰 약점이었던 "완료 버튼만 누르면 기록되는 구조"를 제거한다. 사용자는 실제 타이머를 시작하고, 일시정지/재개할 수 있으며, 타이머가 00:00에 도달한 뒤에만 보상을 받을 수 있다.

## 구현된 것

- 15/25/50분 세션 선택
- 실제 카운트다운 표시
- 집중 시작
- 일시정지
- 다시 시작
- 타이머 초기화
- 타이머 완료 후 보상 받기
- 완료 세션 localStorage 저장
- 오늘 집중 시간 / 완료 횟수 / 포인트 유지
- 타이머 종료 전 보상 비활성화

## 대화

**사용자:** 이제는 바로 완료 버튼을 누를 수 없어서 MVP0보다 훨씬 낫습니다. 하지만 브라우저 탭을 닫거나 새로고침하면 진행 중인 타이머가 이어지나요?

**앱 개발자:** 아직 이어지지 않습니다. 완료된 세션만 localStorage에 저장하고, 진행 중인 세션 상태는 저장하지 않았습니다. MVP0.5의 목표는 "완료 전 보상 차단"과 "실제 카운트다운"이었고, 백그라운드/새로고침 복원은 다음 안정화 과제입니다.

**사용자:** 그러면 사용자가 25분 중 20분을 하고 새로고침하면 날아갑니다. 실제 사용자 테스트에서는 불만이 클 수 있습니다.

**앱 개발자:** 맞습니다. MVP1 전에 `startedAt`, `duration`, `remainingSeconds`, `timerState`를 저장하고, 재접속 시 경과 시간을 계산해야 합니다. 지금 버전은 내부 검증용으로는 충분하지만 공개 베타용으로는 부족합니다.

**사용자:** 일시정지는 들어갔는데, 무제한 일시정지를 허용하면 "진짜 집중 시간"이라는 데이터 신뢰도가 떨어지지 않나요?

**앱 개발자:** 현재는 사용자 신뢰 우선으로 무제한 일시정지를 허용했습니다. 친구 경쟁이 들어가는 MVP2부터는 총 소요 시간, 일시정지 횟수, 실제 집중 인정 기준을 별도로 설계해야 합니다.

**사용자:** 타이머 완료 후 보상 받기를 누르지 않으면 기록이 남지 않습니다. 사용자가 완료 화면을 닫으면 손해 아닌가요?

**앱 개발자:** 맞습니다. 지금은 보상 수령을 명시적으로 만들었지만, 실제 제품에서는 00:00 도달 시 자동 기록하고 보상 화면만 보여주는 방식이 더 안전할 수 있습니다. 다만 초기 검증에서는 "보상 버튼이 언제 활성화되는지"가 명확해서 테스트하기 쉽습니다.

**사용자:** 포인트 무한 획득은 막았지만, 15분 타이머를 여러 번 돌리면 계속 포인트가 쌓입니다. 이건 정상인가요?

**앱 개발자:** 현재는 정상입니다. 완료된 세션마다 +5pt입니다. 다만 포인트 경제가 들어가기 전까지는 포인트를 많이 쌓아도 사용할 곳이 없고, 친구 경쟁 전에는 부정행위 리스크가 낮습니다.

**사용자:** 모바일 사용성은 아직 애매합니다. 캐릭터 영역이 커서 타이머 컨트롤이 아래로 밀릴 수 있습니다.

**앱 개발자:** 맞습니다. 다음 UI 조정에서는 모바일에서 타이머 패널을 먼저 보여주거나, 캐릭터 크기를 줄이는 반응형 레이아웃을 검토해야 합니다.

**사용자:** 이제 MVP1 계정 저장으로 넘어가도 되나요?

**앱 개발자:** 기능상으로는 가능하지만, 그 전에 하나를 더 확인해야 합니다. 진행 중인 타이머 복원 없이 서버 저장으로 넘어가면 서버에 저장할 세션 모델이 불완전해집니다. MVP0.6으로 진행 중 세션 복원과 완료 자동 기록을 먼저 정리하는 편이 더 안전합니다.

## 개선 우선순위

1. 진행 중 타이머 상태를 localStorage에 저장한다.
2. 새로고침/재접속 시 경과 시간을 계산해 타이머를 복원한다.
3. 00:00 도달 시 세션을 자동 기록할지, 보상 버튼을 유지할지 결정한다.
4. 일시정지 횟수와 총 소요 시간을 기록할지 결정한다.
5. 모바일에서 타이머 컨트롤을 더 위로 배치한다.
6. 친구 경쟁 전에 "인정 가능한 집중 세션" 기준을 정의한다.

## 결론

MVP0.5는 MVP0의 가장 큰 제품 리스크였던 즉시 완료/무한 포인트 문제를 줄였다. 그러나 아직 공개 사용자 테스트용은 아니다. 다음 핵심은 친구 기능이 아니라 진행 중 세션의 신뢰성, 즉 새로고침/탭 닫기/백그라운드 상황에서 기록이 얼마나 안전하게 유지되는가다.
