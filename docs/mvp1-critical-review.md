# MVP1 비판적 검토 대화

## 범위

MVP1은 계정 기반 개인 기록 단계다. 목표는 회원가입/로그인 후 완료된 포모도로 기록이 서버에 저장되고, 오늘/이번 주 집중 시간, 기본 캐릭터, 포인트가 계정 단위로 유지되는 것이다. 기존 타이머 UI의 구조와 시각 방향은 유지했다.

## 구현된 것

- 회원가입
- 로그인
- 로그아웃
- httpOnly 세션 쿠키
- 파일 기반 서버 저장소 (`.data/pomochi-db.json`)
- 기본 캐릭터 `불꽃이` 계정별 생성
- 완료된 포모도로 서버 저장
- 서버 저장 후 포인트 +5 누적
- 오늘 집중 시간 / 오늘 완료 횟수 표시
- 이번 주 집중 시간 / 완료 횟수 서버 집계
- 로그인 상태에서 앱 재접속 시 서버 프로필 로드
- 비로그인 상태에서는 기존 localStorage 기록 유지

## 대화

**사용자:** MVP1이면 서버 저장이라고 했는데 PostgreSQL이 아니라 JSON 파일입니다. 이걸 서버 저장이라고 볼 수 있나요?

**앱 개발자:** MVP1 검증 기준에서는 볼 수 있습니다. 브라우저 localStorage가 아니라 API와 서버 프로세스를 거쳐 파일에 저장됩니다. 다만 운영용 저장소는 아닙니다. MVP2나 베타 전에는 PostgreSQL 또는 SQLite 같은 DB로 교체해야 합니다.

**사용자:** 계정 기능이 생기면서 UI가 바뀐 것 아닌가요? UI는 수정하지 말라고 했습니다.

**앱 개발자:** 타이머, 캐릭터, 탭, 버튼의 기존 배치와 스타일은 유지했습니다. 추가된 것은 MVP1에 필요한 최소 계정 입력 영역입니다. 이 기능 없이 회원가입/로그인을 검증할 방법이 없어서 헤더 안에 작은 영역으로만 붙였습니다.

**사용자:** 서버에 저장된다면 새로고침 후에도 기록이 살아야 합니다. 확인했나요?

**앱 개발자:** API와 컴포넌트 테스트에서 서버 프로필을 다시 불러와 points/sessions를 반영하는 흐름을 검증했습니다. 실제 dev 서버에서도 회원가입, 포커스 완료 API, 프로필 조회가 200으로 동작했습니다.

**사용자:** 지금 비로그인 기록과 로그인 기록이 섞일 수 있나요?

**앱 개발자:** 로그인하면 서버 프로필이 화면 상태를 덮어씁니다. 비로그인 localStorage 기록을 자동 업로드하지는 않습니다. 이건 의도적인 보수적 선택입니다. 자동 병합은 중복 기록을 만들 수 있으므로 별도 UX가 필요합니다.

**사용자:** 비밀번호 보안은 충분한가요?

**앱 개발자:** 평문 저장은 하지 않고 PBKDF2 해시를 저장합니다. 하지만 비밀번호 재설정, 이메일 인증, rate limit, CSRF 방어는 아직 없습니다. MVP1 내부 검증에는 충분하지만 공개 서비스 기준은 아닙니다.

**사용자:** 이번 주 집중 시간은 UI에 보이지 않는데 구현했다고 할 수 있나요?

**앱 개발자:** 서버 프로필 API에는 `weekMinutes`, `weekCount`가 집계되어 반환됩니다. MVP1 요구사항은 서버 집계까지이고, 친구 경쟁이 들어가는 MVP2에서 UI 노출이 필요합니다.

**사용자:** 테스트가 실제 브라우저까지 확인하나요?

**앱 개발자:** jsdom 단위 테스트, API 테스트, 빌드, audit에 더해 실제 Chrome 자동화로 회원가입 UI와 타이머 선택/카운트다운을 확인했습니다. 이전에 있었던 클라이언트 번들 404 문제도 static resource 실패가 없는지 확인했습니다.

## 코드에 반영한 검토 결과

- 계정 입력 영역을 `<form>`으로 정리해 브라우저의 password input 경고를 제거했다.
- dev/build 산출물 충돌을 피하기 위해 `next build`만 `.next-build`를 쓰도록 조정했다.
- 서버 저장소, API, UI 동기화 각각에 테스트를 추가했다.

## 남은 리스크

1. JSON 파일 저장소는 운영용이 아니다.
2. 진행 중 타이머 상태는 아직 서버에 저장하지 않는다.
3. 비로그인 localStorage 기록을 로그인 계정으로 병합하지 않는다.
4. 비밀번호 재설정, 이메일 인증, rate limit은 없다.
5. 이번 주 집계는 API에는 있지만 UI에는 아직 노출하지 않는다.
6. MVP2 친구 경쟁 전에 "인정 가능한 집중 세션" 기준을 더 엄격히 정의해야 한다.

## 결론

MVP1의 핵심인 계정 기반 개인 기록은 동작한다. 다만 이 단계는 운영 인프라 완성이 아니라 "서버를 거쳐 계정별 기록을 유지할 수 있는가"를 검증하는 단계다. 다음 단계로 넘어가기 전 가장 중요한 결정은 JSON 파일 저장소를 실제 DB로 교체할 시점과, 비로그인 기록을 계정에 병합할지 여부다.
