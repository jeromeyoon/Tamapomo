# MVP0.5 클릭/타이머 미동작 버그 검토

## 증상

사용자가 집중 시간을 선택해도 `25:00`에서 바뀌지 않고, `집중 시작`을 눌러도 시간이 줄어들지 않았다.

## 원인

컴포넌트 상태 로직 자체가 아니라 dev 서버 산출물 충돌이었다.

검증 과정에서 `next dev` 서버가 켜진 상태로 `next build`를 실행했고, 둘 다 같은 `.next` 디렉터리를 사용했다. 그 결과 production build 산출물이 dev 서버 산출물을 덮어썼고, 브라우저가 요청한 개발용 클라이언트 번들이 404가 됐다.

브라우저에서는 HTML이 서버 렌더링되어 화면은 보였지만, `main-app.js`, `app/page.js` 같은 클라이언트 JS가 로드되지 않아 React hydration이 되지 않았다. 그래서 모든 버튼이 정적인 HTML처럼 보였다.

## 수정

- `next dev`는 `.next-dev`를 사용하도록 분리
- `next build`는 기존 `.next` 사용
- `next.config.mjs`에서 `NEXT_DIST_DIR` 환경변수로 `distDir` 설정
- `package.json`의 dev script에 `cross-env NEXT_DIST_DIR=.next-dev` 추가
- 재발 방지 테스트 추가

## 재검증

실제 Chrome 자동화로 확인했다.

- 초기 타이머: `25:00`
- `15분` 클릭 후: `15:00`
- `집중 시작` 후 약 2초 경과: `14:58`
- dev 서버 실행 중 `next build`를 다시 실행한 뒤에도:
  - `50분` 클릭 후: `50:00`
  - `집중 시작` 후: `49:59`

## 비판적 검토 대화

**사용자:** 테스트는 통과했다면서 왜 실제 화면에서는 클릭이 안 됐나요?

**앱 개발자:** 기존 테스트는 React 컴포넌트를 jsdom에서 직접 렌더링했습니다. 그래서 컴포넌트 내부 상태 로직은 검증했지만, 실제 Next dev 서버가 클라이언트 번들을 제대로 제공하는지는 검증하지 못했습니다.

**사용자:** 그러면 테스트가 부족했던 것 아닌가요?

**앱 개발자:** 맞습니다. 단위 테스트만으로는 부족했습니다. 이번 버그는 컴포넌트 코드보다 dev/build 실행 환경 문제였기 때문에, 실제 브라우저에서 `_next/static` 리소스가 200으로 로드되고 클릭 후 DOM이 바뀌는지 확인해야 했습니다.

**사용자:** 앞으로도 빌드 검증을 하면 dev 서버가 또 깨질 수 있나요?

**앱 개발자:** 이제는 dev와 build 산출물 디렉터리를 분리했기 때문에 같은 방식으로 깨지지 않습니다. dev 서버는 `.next-dev`, production build는 `.next`를 씁니다.

**사용자:** 이 수정은 제품 기능과 직접 관련 없는데 꼭 필요했나요?

**앱 개발자:** 필요했습니다. 사용자가 보는 앱에서 클릭이 안 되면 타이머 기능을 검증할 수 없습니다. MVP0.5의 핵심은 실제 타이머 신뢰성인데, 클라이언트 JS 로드 실패는 그 검증 자체를 무효화합니다.

## 남은 리스크

- 진행 중 타이머를 새로고침 후 복원하는 기능은 아직 없다.
- 실제 모바일 브라우저에서 백그라운드 전환 시 타이머 정확도는 별도 검증이 필요하다.
- jsdom 단위 테스트 외에 브라우저 기반 smoke test를 정식 npm script로 추가할지 결정해야 한다.
