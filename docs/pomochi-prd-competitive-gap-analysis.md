# 뽀모치 현재 구현 / PRD / 경쟁 앱 비교 분석

작성일: 2026-06-21  
역할 관점: 게임 기획자 + 게임 개발자  
목적: 현재 구현을 PRD v2.1 및 경쟁 앱과 비교하고, 다음 수정 우선순위를 결정한다.

---

## 1. 결론 요약

현재 구현은 `MVP-0~MVP-2 초기`까지의 얇은 세로 조각을 갖췄다. 타이머, 포인트, 로컬 기록, 계정 기반 서버 저장, 기본 캐릭터 반응, 기록 달력/그래프, 날짜별 집중 로그, 친구 추가, 오늘/이번 주 친구 랭킹, 테스트가 있다. 최근 수정으로 꾸미기/스토어/배경 아이템을 제거해 포모도로 집중도도 좋아졌다.

PRD의 핵심 문장인 **"양육은 배경, 소셜이 주인공"** 기준으로 보면 이제 친구 경쟁의 최소 단위는 보인다. 다만 아직은 "친구와 함께 만든 관계"보다 "친구 기록을 보는 랭킹"에 가깝기 때문에, Study Bunny / Forest / Focus Plant와 비교했을 때 장기 차별점은 더 검증해야 한다.

따라서 다음 수정의 방향은 캐릭터를 더 꾸미는 것이 아니라:

1. 친구 경쟁 MVP의 사용성을 실제 친구 초대/수락 흐름까지 보강한다.
2. 돌봄 시스템은 포모도로 완료 피드백을 해치지 않게 단순화한다.
3. 기록 화면은 "분석"보다 다음 집중을 유도하는 문구를 계속 강화한다.
4. 캐릭터 성장은 "집중 기록의 시각화"로 제한한다.

## 2. 현재 구현 요약

### 2.1 구현된 핵심 기능

현재 코드 기준:

- `홈 / 기록 / 친구` 3탭 구조.
- 15분, 25분, 50분 기본 포모도로.
- 직접 입력 분 단위 타이머.
- 타이머 시작, 일시정지, 재시작, 리셋.
- 완료 후 `보상 받기 +5pt`.
- 오늘 집중 시간 / 완료 세션 수 표시.
- 로컬 모드: `localStorage`에 포인트와 세션 저장.
- 서버 모드: 회원가입, 로그인, 로그아웃, 세션 쿠키, 완료 기록 API 저장.
- 기록 화면: 달력/막대 그래프, 날짜별 시작 시각, 집중 품질, 행동 유도 문구.
- 기본 캐릭터 `콩냥이` SVG.
- 성장 단계: 씨앗, 새싹, 꽃봉오리, 만개.
- 성장 설정: 테스트용 빠른 값과 운영 기본값 분리.
- 돌봄 게이지: 밥, 기운, 건강, 청결, 애정.
- 돌봄 행동: 밥 주기, 재우기, 약 주기, 씻기, 쓰다듬기, 놀아주기.
- 타이머 종료 시 랜덤 상태 변화: 더러움, 아픔, 지루함.
- 포모도로 완료 피드백: 정상 완료 시 캐릭터 기분 상승, 잦은 일시정지/화면 이탈 시 기분 하락과 배고픔 증가.
- 친구 경쟁 MVP: 이메일 기반 친구 추가, 친구 목록, 오늘/이번 주 랭킹, 앞섬/뒤처짐 요약 문구.

### 2.2 구현에서 제거된 것

최근 방향 전환에 맞게 제거됨:

- 방 배경 테마.
- 머리 액세서리.
- 캐릭터 꾸미기 화면.
- 스토어 탭.
- 꾸미기 구매/저장 로직.

이 제거는 PRD의 "첫 베타 전까지 제외: 꾸미기 스토어"와 일치한다.

## 3. PRD와 현재 구현 비교

| 영역 | PRD 의도 | 현재 구현 | 판단 |
|---|---|---|---|
| 핵심 가치 | 친구와 약속/경쟁하며 집중 | 친구 랭킹 MVP 구현 | 초기 검증 가능 |
| MVP-0 | 타이머, 기록, 기본 캐릭터, 포인트 | 대부분 구현 | 양호 |
| MVP-1 | 계정 기반 개인 기록 | JSON 파일 저장 기반 구현 | 검증용으로 가능, 운영 전 DB 필요 |
| MVP-2 | 친구 추가, 친구 랭킹, 차이 메시지 | 이메일 친구 추가와 기간별 랭킹 구현 | 수락/초대 UX 보강 필요 |
| 돌봄 | 배고픔/행복도 2개만 | 5개 게이지 | 복잡도 과다 |
| 질병 | 첫 베타 전 제외 | 아픔/약 주기 존재 | PRD와 충돌 |
| 꾸미기 | 첫 베타 전 제외 | 최근 제거됨 | 정합 |
| 캐릭터 | 최초 5종 + 리롤 | 콩냥이 1종 | MVP에는 가능, PRD와는 축소 |
| 성장 기간 | 45~60일 장기 애착 | 테스트용/운영 기본값 분리 | 운영 수치 검증 필요 |
| 소셜 | 실시간 없이 약속/랭킹/공동 목표 | 친구 랭킹만 구현 | 약속/공동 목표는 후속 |
| 게임 중독 방지 | 게임은 집중 결과여야 함 | 돌봄 버튼으로 포인트 소비 가능 | 루프 재정의 필요 |

## 4. 경쟁 앱 분석

### 4.1 Study Bunny

Study Bunny는 공부 타이머, 코인, 스토어 아이템/음악, To Do, Flashcards, Study Tracker를 결합한다. 즉, "귀여운 공부 도우미 + 생산성 도구 묶음"이 강점이다.

뽀모치와의 비교:

- 현재 뽀모치는 Study Bunny보다 도구 수가 적다.
- 반대로 도구를 늘리기보다 친구 경쟁과 함께 만든 캐릭터로 차별화해야 한다.
- 지금 단계에서 스토어/꾸미기를 넣으면 Study Bunny의 약한 복제품처럼 보일 위험이 있다.

수정 시사점:

- To Do, Flashcards 같은 학습 도구는 지금 넣지 않는다.
- 대신 "오늘 25분 한 번 더 하게 만드는 이유"를 친구 경쟁으로 만든다.

### 4.2 Forest

Forest는 집중 시간 동안 나무를 키우고, 사용자가 집중을 포기하면 나무가 시드는 구조다. 공식 소개 기준으로 앱 차단, Plant Together, 실제 나무 후원, 집중 분석까지 확장되어 있다.

뽀모치와의 비교:

- Forest는 집중 실패에 대한 상징적 손실이 강하다.
- 뽀모치는 PRD상 "캐릭터는 죽지 않는다"가 철학이다.
- 따라서 Forest처럼 강한 처벌을 따라 하면 제품 철학과 충돌한다.

수정 시사점:

- 실패 페널티보다 "다시 시작" 피드백을 강화한다.
- 친구와의 작은 격차 메시지로 재시작 동기를 준다.
- 앱 차단은 초기 범위에서 제외한다.

### 4.3 Flora

Flora는 친구와 함께 나무를 심고, 진행 상황을 공유하며, daily/weekly/monthly 통계를 제공한다. 공동 집중과 친구 기반 동기 부여가 강하다.

뽀모치와의 비교:

- PRD의 "실시간 없는 함께"는 Flora/Forest의 공동 집중보다 가볍게 구현 가능한 대안이다.
- 현재 뽀모치에는 친구와 함께 한다는 느낌이 아직 없다.

수정 시사점:

- MVP-2에서 친구 랭킹을 먼저 넣는다.
- 실시간 동시 세션보다 "오늘 친구가 몇 분 했는지"를 먼저 보여준다.

### 4.4 Focus To-Do

Focus To-Do는 포모도로와 태스크, 프로젝트, 통계, 멀티 플랫폼 동기화가 강점이다. 생산성 앱으로서의 완성도가 경쟁력이다.

뽀모치와의 비교:

- 뽀모치가 순수 생산성 경쟁으로 가면 불리하다.
- 현재 기록 달력/그래프는 좋은 기반이지만, 태스크 관리까지 확장하면 범위가 커진다.

수정 시사점:

- 당장은 태스크 관리보다 완료율, 재시작률, 친구 경쟁 참여율을 측정한다.
- 기록 화면은 "분석"보다 "오늘 다시 시작"을 돕는 문구를 추가하는 쪽이 맞다.

### 4.5 Focus Plant

Focus Plant는 집중 시간을 물방울로 바꾸고, 황무지를 식물로 채우는 수집/복원 메타를 제공한다. 사용자 수와 장기 콘텐츠를 강조한다.

뽀모치와의 비교:

- 현재 뽀모치의 콩냥이 성장은 매력적이지만, 장기 콘텐츠 레이어는 없다.
- 다만 지금 장기 수집을 먼저 만들면 포모도로 핵심 검증이 늦어진다.

수정 시사점:

- 장기 컬렉션은 Phase 2 이후.
- 지금은 콩냥이 성장 단계가 "집중 누적의 피드백"으로 보이면 충분하다.

### 4.6 Finch / Habitica

Finch는 자기 돌봄 행동으로 펫을 돌보는 앱이고, Habitica는 현실 할 일을 RPG로 바꾸는 앱이다. 둘 다 "행동 완료 → 캐릭터/게임 보상" 구조가 강하다.

뽀모치와의 비교:

- Finch처럼 캐릭터 애착을 줄 수 있지만, 뽀모치의 핵심 행동은 self-care가 아니라 집중이다.
- Habitica처럼 게임 규칙을 많이 넣으면 중학생~대학생 라이트 사용자에게 부담이 커질 수 있다.

수정 시사점:

- 캐릭터 돌봄 액션 수를 줄인다.
- 포모도로 완료가 가장 강한 돌봄 행동이 되게 한다.

## 5. 가장 큰 제품 리스크

### 리스크 1. 타겟 사용자 톤 관리

PRD의 주 타겟은 `중학생~대학생`으로 확정했다. 이 결정으로 친구 경쟁, 이메일 로그인, 15/25/50분 포모도로, 기록 분석 UI는 제품 방향과 맞는다.

남은 위험:

- 캐릭터 문서의 귀여운 톤이 너무 유아적으로 흐르면 중고등학생/대학생에게 부담이 될 수 있다.
- 경쟁 문구가 과하면 공부 스트레스 앱으로 느껴질 수 있다.
- 대학교 사용자는 과제/시험/자격증 맥락이 필요하고, 중학생은 더 짧은 시작 문구가 필요하다.

수정 방향:

- 디자인은 귀엽게 유지하되 문구는 "아이용"보다 "친구와 공부하는 플래너" 톤으로 쓴다.
- 랭킹은 압박보다 재시작 유도에 집중한다.
- 기본 15/25/50분은 유지하고, 짧은 직접 입력은 계속 제공한다.

### 리스크 2. 돌봄 복잡도가 PRD보다 높음

현재는 5개 게이지와 6개 돌봄 행동이 있다. PRD는 배고픔/행복도 2개만 제안한다.

현재 구조의 문제:

- 포모도로보다 돌봄 버튼이 더 눈에 띌 수 있다.
- 사용자가 "해야 할 일"이 늘어난다.
- 게임 중독 방지 원칙과 충돌한다.
- 아이 대상이면 관리 부담이 커지고, 성인 대상이면 귀찮아질 수 있다.

수정 방향:

- MVP에서는 `배고픔`, `기분` 2개만 남긴다.
- `씻기`, `약 주기`, `재우기`는 상태 이펙트용으로 잠시 숨긴다.
- `쓰다듬기`는 무료/무보상 감정 반응으로만 둔다.

### 리스크 3. 소셜 차별점이 아직 얕음

PRD의 가장 큰 차별점은 친구 집중시간 경쟁과 함께 만든 캐릭터다. 현재는 친구 추가와 기간별 랭킹이 동작하지만, 아직 수락/거절, 친구 코드, 약속 세션, 공동 목표는 없다.

수정 방향:

- 다음 소셜 작업은 친구 요청 수락/거절과 초대 링크/코드다.
- 실시간, 채팅, 공동 알은 미룬다.
- 친구 경쟁 조회 후 추가 포모도로 시작률을 측정한다.

### 리스크 4. 운영 성장 수치 검증 필요

테스트용 빠른 성장값과 운영 기본값은 분리했다. 테스트에서는 빠른 값(`0.2/1/2/4/8분`)을 쓰고, 운영 기본값은 `25/60/180/420/900분`으로 둔다.

남은 위험:

- 운영 기본값이 실제 학생 사용자의 1~2주 유지율에 맞는지 아직 검증되지 않았다.
- 장기 성장 목표가 너무 멀면 캐릭터 피드백이 약해질 수 있다.

수정 방향:

- 베타에서는 성장 단계 도달률과 7일 유지율을 같이 본다.
- 운영값은 환경 설정으로 유지해 실험 가능하게 둔다.

### 리스크 5. 서버 저장소가 운영용 DB가 아님

현재 서버 저장은 `.data/pomochi-db.json` 파일 기반이다. MVP 검증에는 충분하지만 친구 경쟁과 랭킹 집계가 들어가면 파일 저장소는 곧 한계가 온다.

수정 방향:

- MVP-2 이전에 SQLite 또는 PostgreSQL로 이동한다.
- 친구 랭킹 쿼리 기준으로 스키마를 먼저 잡는다.

## 6. 수정 우선순위

### P0. 이번 수정에서 정리된 것

1. **PRD 타겟 확정**
   - 주 타겟을 중학생~대학생으로 정리했다.
   - 캐릭터 문서는 귀엽지만 유아적이지 않은 학습 플래너 마스코트 톤으로 조정했다.

2. **성장 수치 환경 분리**
   - `lib/growth-config.ts`로 테스트용 빠른 값과 운영 기본값을 분리했다.
   - 테스트 환경은 빠른 성장값, 운영 기본은 25분 부화와 장기 성장값을 사용한다.

3. **친구 경쟁 MVP**
   - 이메일 기반 친구 추가, 친구 목록, 오늘/이번 주 랭킹, 차이 메시지를 구현했다.

4. **포모도로 완료 후 피드백 개선**
   - 정상 완료 시 캐릭터 기분 상승 문구를 표시한다.
   - 잦은 일시정지/화면 이탈 시 기분 하락과 배고픔 증가 피드백을 넣었다.

5. **기록 화면 행동 유도**
   - 날짜별 집중 시작 시각, 집중 품질, 다음 집중 유도 문구를 추가했다.

### P1. 다음 구현으로 권장

1. **돌봄 게이지 단순화 결정**
   - PRD대로 `배고픔/기분`만 남길지 결정.
   - 지금 유지한다면 PRD를 수정해야 한다.

2. **친구 요청 UX 보강**
   - 이메일 즉시 친구 추가는 MVP 검증용이다.
   - 실제 서비스 전에는 요청/수락/거절, 차단, 초대 링크/코드가 필요하다.

3. **운영 DB 전환**
   - JSON 파일 저장은 로컬 MVP에는 충분하지만 친구 경쟁 운영에는 부족하다.
   - SQLite 또는 PostgreSQL로 랭킹 집계를 옮긴다.

4. **분석 문구 고도화**
   - 현재 행동 유도 문구는 오늘 총합 기준이다.
   - 다음에는 어제 대비, 친구 격차, 자주 시작하는 시간대 기준으로 문구를 바꾼다.

### P2. 베타 전까지 보류

- 꾸미기/스토어.
- 배경 아이템.
- 복잡한 질병.
- 공동 알/번식.
- 가족 계보.
- 실시간 WebSocket.
- 전체 리더보드.
- 프리미엄 구독.
- 앱 차단.

## 7. 추천 수정안: 다음 스프린트 범위

### 목표

`친구 경쟁이 보이는 포모도로 앱`에서 `친구와 함께 계속하게 되는 포모도로 앱`으로 이동한다.

### 포함

- 친구 요청/수락/거절 UX.
- 친구 초대 링크 또는 코드 생성.
- 친구 삭제/차단 최소 기능.
- JSON 파일 저장소에서 SQLite 또는 PostgreSQL로 전환 검토.
- 친구 경쟁 조회 후 포모도로 시작률 이벤트 설계.
- 기록 화면 행동 유도 문구를 친구 격차와 어제 대비 기준으로 확장.

### 제외

- 채팅.
- 실시간 동시 타이머.
- 약속 세션.
- 공동 목표.
- 공동 알.
- 꾸미기/스토어.

### 완료 기준

- 친구 요청을 보내고 수락/거절할 수 있다.
- 친구 아닌 사용자의 랭킹은 보이지 않는다.
- 친구 경쟁 화면을 본 뒤 바로 포모도로를 시작할 수 있는 문구/CTA가 있다.
- 친구 경쟁 관련 테스트가 API, store, UI 레벨에 있다.
- 기존 솔로 타이머 테스트가 깨지지 않는다.

## 8. 게임 기획자 관점 피드백

현재 콩냥이는 귀엽고 앱의 감정 중심으로 충분히 쓸 수 있다. 하지만 캐릭터 자체가 더 복잡해지면 제품이 Study Bunny 쪽으로 끌려간다. 뽀모치가 이기려면 캐릭터는 "내 집중과 친구 관계의 증거"가 되어야 한다.

좋은 방향:

- 포모도로 완료 → 콩냥이 반응.
- 연속 집중 → 콩냥이 성장.
- 친구와 둘 다 집중 → 캐릭터끼리 인사.
- 공동 목표 달성 → 공동 알.

피해야 할 방향:

- 포인트로 아이템 구매하기.
- 방 꾸미기 수집.
- 게이지 관리 게임화.
- 포모도로와 무관한 미니게임.

## 9. 개발자 관점 피드백

현재 코드의 좋은 점:

- 테스트가 핵심 UI와 API 흐름을 잡고 있다.
- `next build`가 통과한다.
- localStorage 모드와 서버 모드가 나뉘어 있어 MVP 검증이 쉽다.
- 타이머/캐릭터/기록이 한 화면 안에서 실제로 동작한다.
- 친구 관계와 랭킹이 store/API/UI 테스트로 묶여 있다.
- 성장 임계값이 테스트/운영 설정으로 분리되어 있다.

수정이 필요한 점:

- `components/pomochi-app.tsx`가 커지고 있다. 다음 기능 전에 `TimerPanel`, `CharacterPanel`, `CarePanel`, `AuthPanel`, `BottomNav`, `FocusHistory`로 분리하는 것이 좋다.
- 친구 기능 운영 전에 저장소 인터페이스를 JSON 파일 전용에서 DB 교체 가능한 형태로 정리해야 한다.
- 캐릭터 데이터는 `lib/characters.ts`에 있지만 실제 메인 캐릭터 렌더링은 컴포넌트 내부에 있어 데이터와 뷰가 분리되어 있지 않다.
- 돌봄 상태 저장은 로컬에만 있고 서버 캐릭터 상태와 연결되어 있지 않다.
- 친구 추가가 현재는 즉시 양방향 연결이라 운영 서비스 전 요청/수락 모델이 필요하다.
